# spotify-collab-graph

[![CI](https://github.com/Ranjith200228/spotify-collaboration-network/actions/workflows/ci.yml/badge.svg)](https://github.com/Ranjith200228/spotify-collaboration-network/actions/workflows/ci.yml)

A leak-safe graph-ML rewrite of the original
`Spotify_Collaboration_Network_Analysis` notebook. Builds an artist
collaboration network from ~158 K track-artist rows, detects communities with
Louvain, and runs a five-way feature ablation to test whether **graph-position
features improve popularity prediction beyond audio features**.

All numbers in this README come from
[`reports/`](reports/) JSON files generated by `scripts/run_analysis.py` —
none are hand-edited.

![Top-PageRank artists in the largest connected component, coloured by Louvain community](docs/network_hero.png)

## Why this rewrite exists

The original notebook reported `R² = 1.00` on popularity prediction from two
graph features — a result too good to be true, and indeed it was. The
project had **two compounding leaks**:

1. **Self-loops from duplicated rows.** The dataset has one row per
   `(track, artist, genre)`; tracks filed under multiple genres produced
   multiple rows per `(track, artist)`. Edge construction iterated those
   rows naively, so the same artist got paired with itself → self-loops →
   `G.neighbors(node)` included the node itself.
2. **`avg_neighbor_popularity` was computed over every neighbour, including
   test-set artists.** After `train_test_split`, the feature for any node
   already contained signal from neighbours that were held out — labels
   leaked into features through the graph.

Together these inflate R² toward 1.0 on a problem where realistic R² is
~0.5. The rewrite fixes both, locks the fix in with a regression test, and
honestly re-runs the ablation. Full breakdown:
[`docs/methodology.md`](docs/methodology.md).

## Does the graph encode genre? (real numbers from `reports/community_metrics.json`)

| Metric                              | Value  |
| ----------------------------------- | ------ |
| Louvain modularity                  | 0.925  |
| Adjusted Rand Index (ARI) vs genre  | 0.129  |
| Normalised Mutual Info (NMI) vs genre | 0.618 |
| Communities detected                | 12,610 |

**Honest interpretation.** Modularity is high largely because half the
graph is singletons, each a perfect "community" of one — that inflates the
score and shouldn't be read in isolation. The more meaningful signals:

- **NMI = 0.62** says topology alone recovers a large fraction of the genre
  information — yes, the collaboration graph encodes genre structure.
- **ARI = 0.13** says the partitions don't line up *cluster-for-cluster* with
  genres — Louvain produces many more (~12.6 K) and finer communities than
  there are genres (114). The communities are sub-genre / scene-level, not
  genre-level.

Top-purity communities give the most readable picture:

| Community | Size | Dominant genre   | Purity |
| --------- | ---- | ---------------- | ------ |
| 257       | 414  | funk             | 59.4 % |
| 4336      | 336  | hardstyle        | 57.7 % |
| 728       | 610  | opera            | 56.2 % |
| 1894      | 600  | drum-and-bass    | 51.3 % |
| 668       | 541  | minimal-techno   | 30.5 % |

Opera, hardstyle, funk, and drum-and-bass collaborators stick together
tightly; broader bins (`progressive-house`, `pop-film`) are messier because
they overlap multiple scenes.

## Do graph features beat audio features? (real numbers from `reports/model_ablation.json`)

Single 80/20 node split, `random_state=42`, `GradientBoostingRegressor(n_estimators=200, max_depth=3)`.

| Model           | MAE   | RMSE  | R²     |
| --------------- | ----- | ----- | ------ |
| `baseline_mean` | 14.43 | 17.52 | -0.000 |
| `audio_only`    | 12.81 | 16.02 |  0.164 |
| `genre_only`    | 11.05 | 14.39 |  0.325 |
| `graph_only`    |  8.77 | 12.84 |  0.463 |
| **`combined`**  | **7.69**  | **11.41** | **0.576** |

**Headline: `lift_graph_over_audio`** (from the JSON)

| Metric                     | Value |
| -------------------------- | ----- |
| RMSE, `audio_only`         | 16.02 |
| RMSE, `combined`           | 11.41 |
| Absolute RMSE improvement  | 4.61  |
| Relative RMSE improvement  | **28.8 %** |
| MAE, `audio_only`          | 12.81 |
| MAE, `combined`            | 7.69  |
| Absolute MAE improvement   | 5.12  |

**Reading.** `graph_only` alone (R² = 0.46) **beats `audio_only` (R² = 0.16)
by a wide margin**. Knowing who an artist collaborates with — and where
they sit in the network — predicts popularity better than the musical
fingerprint does. `combined` adds the best of both and reaches R² = 0.58,
a 28.8 % RMSE reduction over audio alone.

Crucially, none of the R² scores approach 1.0; the leakage that produced
the original notebook's bogus perfect fit is gone, and the
`test_neighbor_pop_uses_only_train_nodes` regression test fails CI the
moment anyone reintroduces it.

## Reproducibility

```bash
python -m venv .venv
# Windows PowerShell:
.venv\Scripts\Activate.ps1
# bash / WSL:
source .venv/bin/activate

pip install -r requirements.txt

# Place SpotifyFeatures.csv at data/raw/SpotifyFeatures.csv,
# or run the HuggingFace fallback download:
powershell -ExecutionPolicy Bypass -File scripts/fetch_data.ps1

python scripts/run_analysis.py
pytest -v
ruff check src tests scripts
```

The whole pipeline runs in ~2 minutes on the ~30 K-node graph; outputs are
deterministic given the default `random_state=42`.

## Limitations and honest caveats

- **The dataset is HuggingFace's `maharshipandya/spotify-tracks-dataset`,
  not the Kaggle "Ultimate Spotify Tracks DB" the original notebook used.**
  The Kaggle file isn't on a public mirror, and the loader supports both
  schemas — but the absolute numbers in `reports/` are tied to the
  HuggingFace dump (~158 K rows, 29.8 K artists, 114 genres). Dropping the
  Kaggle CSV at `data/raw/SpotifyFeatures.csv` and re-running gives a
  different-but-comparable set of numbers; the *story* (graph features
  helpful, R² ~0.5) should generalise.
- **Most artists in this dataset are solo.** Only ~66 % have any
  collaborator at all, and the densest interesting structure lives inside
  the largest connected component (41 % of the graph). Graph features for
  isolated nodes are `NaN` and get train-median-imputed.
- **Single split, no cross-validation.** A more rigorous result would
  cross-validate the ablation; the current pipeline reports one held-out
  test fold. The point of the rewrite was honesty about leakage, not
  squeezing the last point of R².
- **No causal claims.** Graph position correlates with popularity. We do
  not claim that collaborating more *makes* you popular — popular artists
  attract collaborators, and reverse causation is plausible.

## Future improvements

- k-fold cross-validation across the ablation
- Node2Vec / DeepWalk embeddings as an alternative to hand-engineered
  structural features
- Genre-stratified split to test whether the lift survives within-genre
  prediction
- Incorporate release dates / temporal edges (collab order matters for any
  "what comes next" question)
- Hyperparameter tuning of GradientBoostingRegressor (currently
  fixed-budget defaults)
